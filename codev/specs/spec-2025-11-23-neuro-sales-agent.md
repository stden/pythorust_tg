# Specification: Нейропродажник (универсальный продажник с обучением)

## Metadata
- **ID**: spec-2025-11-23-neuro-sales-agent
- **Status**: draft
- **Created**: 2025-11-23

## Goal & Positioning
Универсальный продажник для Telegram (чат-first, голос как опция), который знает ассортимент/цены/остатки, ведёт диалог по продажным скриптам, сам доучивается на реальных переписках/звонках и поддерживает разные вертикали без ручной перепрошивки промптов.

## Problem Statement
- Продажи держатся на ручных операторах: медленный ответ, разрыв в скриптах, нет единого каталога и контроля качества.
- Скрипты разъезжаются между нишами, обучение новых операторов дорого; история и контакты теряются, лиды плохо фиксируются.
- Нет системы непрерывного обучения: фидбек из чатов не превращается в улучшения промпта/знаний, метрики конверсий и реакций не собираются.

## Current State
- Базовый автответчик (`autoanswer.py`) без продажных скриптов и каталога, без CRM/Sheets логирования лидов.
- Телеграм-чтение (`read.py`, `tg.py`) без триггеров продаж и CTA.
- Голоса нет в проде: нет SIP-подключения и стабильного STT/TTS пайплайна (есть заготовки в `integrations/`).
- Каталог и цены живут вручную (листы/сообщения), нет источника правды и проверки наличия.

## Desired State
- Chat-first AI-продавец отвечает ≤3с p95, инициирует офферы по триггерам (новый подписчик, вопрос про цену, молчание >N мин).
- Знает ассортимент/цены/остатки/акции; делает апсейл/даунсейл/комплекты; при неизвестных данных отвечает «уточняю» и ставит эскалацию.
- Фиксирует лиды/сделки в CRM/Google Sheet с контактами, суммой, ссылкой на диалог, статусом; исключает дубликаты.
- Логи и метрики (конверсии, скорость ответа, эскалации, отказы) сохраняются и видны владельцу.
- Эскалация человеку по стоп-словам, пустому каталогу, задержке ответа, N неудачных попытках оплаты/доставки.
- Профиль настраивается per-чат (тон, агрессия, язык, ниша, скрипт), можно быстро сменить вертикаль без переписывания кода.
- Обучение: новые скрипты/каталог/разбор чатов автоматически попадают в knowledge base; доступен offline A/B и auto-eval на эталонных диалогах.
- Голос (MVP-опция): SIP + STT→LLM→TTS, отклик ≤1.5с p95, единая логика с чатами.

## Stakeholders
- **Primary**: владельцы/операторы продажных чатов; конечные покупатели.
- **Secondary**: руководители продаж/маркетинга (метрики, лиды).
- **Tech**: разработчик/DevOps проекта (`my_telegram` репо).

## Scope
- In scope: чат-бот продажник с каталогом и CTA; логирование лидов; обучение на чате/каталоге; конфиг per-чат; база знаний для скриптов; опциональный голосовой воркер.
- Out of scope (MVP): кассовые платежи внутри Telegram, авто-дозвон по базе без согласия, омниканальность (WA/VK) кроме Telegram/SIP.

## Success Criteria
- [ ] p95 время первого ответа в чате ≤3с; в голосе round-trip ≤1.5с.
- [ ] ≥90% ответов содержат CTA или явный следующий шаг.
- [ ] 0 галлюцинаций по цене/наличию (валидация по каталогу); при пробеле — «уточняю» + эскалация.
- [ ] ≥95% лидов с покупательским намерением записаны в CRM/Sheet с ссылкой на диалог.
- [ ] Обучение: новая позиция/цена в каталоге появляется в ответах без перезапуска; offline auto-eval ≥80% pass на тестовом наборе.
- [ ] Логи/метрики доступны владельцу (конверсии, скорость, эскалации, отказ/галлюцинации).
- [ ] Все тесты по логике бота проходят; нагрузочный тест укладывается в Performance Requirements.

## Constraints
### Technical
- Python 3 + Telethon; LLM через `integrations/openai_client.py` или `integrations/ollama_client.py`.
- Источник правды для каталога: Google Sheet/CSV/Notion API (выбираем 1, остальные синхронизируются).
- .env для ключей; нельзя логировать секреты/PII; личные данные маскировать в логах.
- Голос: SIP провайдер (Zadarma/Exolve/Twilio) + локальный STT/TTS (Whisper/Silero) для контроля стоимости.

### Business
- Соответствие ToS Telegram и SIP-провайдера; без спама и холодных рассылок.
- Двуязычие RU/EN минимум; агрессивность продаж регулируется.
- Стоимость: локальные модели предпочтительны при стабильном качестве.

## Assumptions
- Каталог доступен в структурированном виде и обновляется ≥1 раз/сутки.
- Ключи OpenAI или локальная LLM готовы; SIP ключи есть для теста.
- Есть доступ к Google Sheet/CRM для записи лидов.
- Разрешено использовать автоответчик в целевых чатах; есть оператор для эскалаций.

## Functional Requirements
### Core Sales Behavior
- Ответ ≤3с p95; короткие реплики, всегда CTA (оплата/доставка/контакт/следующий шаг).
- Персонализация per-чат: имя бота, тон, язык, агрессия/срочность, лимиты скидок, ниша.
- Триггеры: новое сообщение/подписка, молчание >N минут, вопрос о цене/наличии, просмотр каталога, брошенный диалог.
- Эскалация: стоп-слова, отсутствие данных, 2+ неудачных попыток оплаты/адреса, долгий ответ LLM.

### Catalog & Knowledge
- Интеграция с одним источником правды (Sheet/CSV/Notion); кэш + TTL обновлений.
- Поля: SKU/название, категория, цена, наличие/остаток, акции/скидки, связки для апсейл/комплектов.
- Каталог используется для валидации цены/наличия; при расхождении — «уточняю» + эскалация.
- База скриптов: приветствие, квалификация, возражения, закрытие, follow-up. Хранится в файловом/kv-хранилище, версионируется.

### Learning Loop (обучение)
- Логи диалогов + метки исхода (лид/нет, причина отказа, эскалация) сохраняются.
- Автосбор датасета: успешные/неуспешные реплики, обработка возражений, неизвестные SKU.
- Offline auto-eval по эталонным диалогам перед выкладкой новой версии промпта/модели.
- Поддержка A/B профилей промпта; метрики конверсии/скорости сравниваются.
- Обновление знаний без перезапуска: обновление каталога/скриптов подтягивается через watcher (пуллинг или webhook).

### Lead & CRM Flow
- Поля лида: имя/username, контакт (телефон/ник), запрос/товар, сумма/валюта, статус (new/qualified/won/lost/escalated), ссылка на диалог/сообщение.
- Дедупликация лидов по chat_id + username/phone; обновление статуса вместо дубля.
- Синк в Google Sheet/CRM через сервисный аккаунт; ретраи + dead-letter для неудачных записей.

### Voice (опция MVP)
- SIP входящие/исходящие, IVR минимальный (приветствие → оператор → AI).
- STT (Whisper/Silero), нормализация текста (мат/шумы), LLM ответ, TTS (Silero/Yandex) → RTP.
- Таймауты и барьер тишины; быстрая эскалация на человека по стоп-словам/задержке.

### Operations & Admin
- Команды оператора: включить/поставить на паузу чат, сменить профиль/скрипт, тестовый прогон диалога, выгрузка логов/метрик.
- Фича-флаги: включение голоса, A/B профилей, автоэскалации, автофоллоу-ап.

## Non-Functional / Performance
- **Latency**: чат ≤3с p95; голос round-trip ≤1.5с p95.
- **Throughput**: 10 параллельных чат-диалогов/инстанс; 3 голосовых на GPU 12GB.
- **Availability**: 99% uptime; graceful restart без потери state (state в Redis/SQLite).
- **Cost**: лимиты на токены/минуты STT/TTS; fallback на локальные модели при превышении бюджета.
- **Reliability**: ретраи записи лидов; circuit breaker для внешних API; наблюдаемость (tracing + метрики).

## Data & Integrations
- Каталог: Sheet/CSV/Notion → loader + валидация → кэш/embeddings (для поиска).
- CRM/Sheets: сервисный аккаунт, ретраи; логирование ошибок без PII.
- Модели: OpenAI или локальная LLM; возможность прогрева и k-shot шаблонов.
- Хранилище логов: локально/SQLite → возможный экспорт в S3/Sheets; маскирование PII.

## Architecture Outline
- Telethon listener → Event Router → Dialogue Engine.
- Dialogue Engine: state machine + policy rules (эскалации/CTA) + LLM client + prompt templates.
- Knowledge Service: Catalog adapter + Script store + Embedding search (для уточнений и возражений).
- Lead Service: CRM/Sheet writer + dedup + retry queue.
- Learning Pipeline: логгер диалогов → dataset builder → auto-eval runner → deploy new prompt/profile.
- Voice Worker (опция): SIP endpoint → STT → Dialogue Engine → TTS → RTP.

## Observability & Metrics
- Метрики: p50/p95 latency, конверсия в лид/оплату, эскалации, попытки апсейла, отказ/галлюцинации, ошибки CRM/каталога.
- Логи: вход/выход бота (маскированы контакты), причины эскалаций, изменения профиля/скриптов.
- Алёрты: деградация latency, рост ошибок CRM/каталога, пустой каталог, недоступность LLM/STT/TTS.

## Test Scenarios
### Functional
1. Запрос цены/наличия из каталога → точный ответ + CTA на оплату/доставку.
2. Неизвестный SKU → «уточняю», эскалация, запись проблемы в лог обучения.
3. Добавлена новая позиция в Sheet → бот использует её без перезапуска.
4. Брошенный диалог (молчание >N мин) → follow-up с оффером/доставкой.
5. Возражение по цене → даунсейл/скидка в рамках лимита; если лимит исчерпан — эскалация.
6. Голос: входящий звонок → STT → LLM → TTS ≤1.5с p95, по стоп-слову перевод на оператора.

### Non-Functional
1. Нагрузочный: 10 параллельных диалогов → p95 ≤3с.
2. Failover: LLM недоступен → запасной профиль шаблонных ответов + эскалация.
3. Безопасность: prompt injection не меняет system prompt; PII не утекает в логи/ответы.
4. Learning: auto-eval на эталонном наборе ≥80% перед выкладкой нового профиля.

## Open Questions
- Источник правды каталога: Sheet vs CSV vs Notion; формат и частота обновлений.
- SIP провайдер и тариф (или исключаем голос из первой версии?).
- Мульти-язычие: RU/EN автодетект или ручной флаг?
- Политика скидок/апсейла: жёсткие правила vs LLM-решение, лимиты на день/пользователя.
- Минимальный объём логирования: полные тексты vs метаданные; где хранить долгосрочно.
- Требуются ли антиспам-фильтры/скоринг входящих лидов.

## Risks and Mitigation
| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|------------|
| Галлюцинации цены/наличия | Medium | High | Жёсткая валидация по каталогу, fallback «уточняю», эскалация. |
| Высокая задержка LLM/STT/TTS | Medium | High | Прогрев, короткий контекст, кэш, локальные модели, timeouts + fallbacks. |
| Блокировка/спам | Low | Medium | Ограничить чаты, соблюдать ToS, антиспам, ручное подтверждение исходящих. |
| Утечка PII | Medium | High | Маскирование в логах, ACL к логам, ретенция. |
| Качество обучения (плохой датасет) | Medium | Medium | Чистка датасета, auto-eval, A/B, ручное ревью выборок. |
